How Magisk Works: A Deep Dive into Android Rooting

How Magisk Works: A Deep Dive into Android Rooting

Introduction

Magisk has become the de-facto standard for rooting Android devices. Unlike older methods that directly modified the /system partition, Magisk pioneered a systemless approach — leaving the system partition completely untouched while still granting full superuser access. But how does it actually work under the hood?

In this post, we’ll break down Magisk’s architecture, from boot image patching to its module system, and understand why it remains undetected by SafetyNet and Play Integrity.

The Problem with Traditional Rooting

Traditional root methods like SuperSU worked by:

  1. Remounting /system as read-write
  2. Placing the su binary in /system/xbin/
  3. Installing a management APK

This had major drawbacks:

  • OTA updates would fail — the modified system partition couldn’t pass verification
  • SafetyNet tripped instantly — Google could detect system modifications
  • dm-verity on newer devices would refuse to boot with a modified system
1
2
3
4

# Traditional root — modifies /system directly
mount -o remount,rw /system
cp su /system/xbin/su
chmod 6755 /system/xbin/su

Magisk solved all of this with one elegant idea: don’t touch /system at all.

Magisk’s Systemless Architecture

Boot Image Patching

Magisk operates entirely within the boot image (boot.img). Here’s what happens when you install Magisk:

  1. Extract the boot image from your device
  2. Unpack it — the boot image contains a kernel (zImage) and a ramdisk (initramfs)
  3. Patch the ramdisk — Magisk injects its init binary (magiskinit) into the ramdisk
  4. Repack and flash the modified boot image

# Simplified boot image structure
boot.img
├── header
├── kernel (zImage)
├── ramdisk (initramfs)
│   ├── init              ← replaced by magiskinit
│   ├── magisk            ← Magisk daemon binary
│   └── overlay.d/        ← Magisk overlay files
└── dtb (device tree)

The critical insight is that the kernel and ramdisk are loaded into RAM at boot — they don’t live on the /system partition. So modifying the boot image gives Magisk execution control without touching /system.

SELinux Policy Patching

One of Magisk’s most powerful features is live SELinux policy patching. Android enforces SELinux in enforcing mode, which would normally block any unauthorized process from running as root.

magiskinit patches the SELinux policy in memory before the real init loads it:

1
2
3
4

// Simplified — what magiskinit does
sepol_allow("untrusted_app", "magisk", "unix_stream_socket", "connectto");
sepol_allow("magisk", "magisk", "process", "setcurrent");
sepol_allow("magisk", "zygote", "process", "ptrace");

This grants the Magisk daemon the permissions it needs without modifying any on-disk policy files.

The Su Binary and Root Management

When an app requests root access:

  1. The app calls the su binary (stored in Magisk’s tmpfs mount)
  2. su connects to the Magisk daemon (magiskd) via a Unix domain socket
  3. magiskd checks the policy in the Magisk Manager app’s database
  4. If approved, magiskd forks a new process with UID 0 and returns a root shell
1
2
3

App (uid=10XXX) → su → magiskd (uid=0) → fork() → root shell
                   ↑                          │
                   └──── Unix socket ─────────┘

The key security model: apps never directly gain root. Everything goes through the daemon, which enforces per-app policies set by the user.

MagiskHide and Zygisk

The Problem: Detection

Banking apps, Google Pay, and some games use SafetyNet / Play Integrity to detect rooted devices. They check for:

  • Modified system properties (ro.debuggable, ro.secure)
  • Presence of root-related files (/sbin/su, /system/app/Superuser.apk)
  • Unusual SELinux states
  • Boot image verification status

MagiskHide (Legacy)

MagiskHide worked by:

  1. Monitoring the Zygote process (parent of all Android apps)
  2. When a target app was about to start, unmounting all Magisk-related bind mounts from that process’s mount namespace
  3. Temporarily reverting system properties

Zygisk (Modern)

Zygisk is Magisk’s newer approach — it loads directly into the Zygote process itself:

1
2
3
4
5
6

Zygote Process
├── Zygisk loaded (libzygisk.so)
│   ├── DenyList: Hide root from listed apps
│   └── Modules: Load Zygisk modules (LSPosed, etc.)
├── fork() → Banking App (clean namespace, no traces)
└── fork() → Normal App (has root access if granted)

By running inside Zygote, Magisk can intercept fork() calls and ensure that denied apps get a completely clean process environment.

The Module System

Magisk modules work through bind mounts — overlaying files on top of the existing system without modifying it:

# Module structure
/data/adb/modules/my_module/
├── module.prop          # Module metadata
├── system/
│   ├── etc/
│   │   └── hosts        # Custom hosts file
│   └── framework/
│       └── services.jar # Modified framework (careful!)
├── post-fs-data.sh      # Runs before Zygote
└── service.sh           # Runs on boot complete

During boot, Magisk scans all modules and creates bind mounts:

1
2

# What Magisk does for each module file
mount -o bind /data/adb/modules/my_module/system/etc/hosts /system/etc/hosts

From the OS perspective, /system/etc/hosts appears to be the original file, but it’s actually the module’s version. The real file underneath is untouched.

Why Magisk Survives OTA Updates

Since Magisk only modifies the boot image:

  1. OTA updates replace /system, /vendor, etc. — Magisk is unaffected because it doesn’t live there
  2. If the OTA replaces the boot image, Magisk needs to be re-patched — but the Magisk app can do this automatically using the “Install to Inactive Slot” feature on A/B partition devices

After OTA:
Slot A:             patched boot.img (Magisk)
Slot B (inactive):  new stock boot.img → Magisk patches this → new patched boot.img

Reboot:
Slot B (active):    new patched boot.img (Magisk) ← still rooted!

Key Takeaways

Aspect Traditional Root Magisk
Modifies /system ✅ Yes ❌ No
Survives OTA ❌ No ✅ Yes (A/B)
SafetyNet bypass ❌ No ✅ Yes (Zygisk)
Module system ❌ Limited ✅ Powerful
SELinux handling ❌ Disabled ✅ Policy patching

Magisk’s genius lies in its systemless philosophy — achieving full root access and system modification capabilities without ever writing to the system partition. By operating from the ramdisk and using bind mounts, it maintains a clean system while providing unparalleled flexibility through its module framework.
